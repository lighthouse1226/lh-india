From AngularJS to Angular – Upgrading Made Simple With UpgradeModule and WebPack

Jon Raymer Development

With the release of Angular (2 and now 4), developers building or maintaining AngularJS (1.x) applications may feel left behind. Angular is a paradigm shift for the framework, in not only the framework design itself, but also by being written natively in TypeScript. With this paradigm shift in mind, the thought of upgrading a large AngularJS app to Angular sounds daunting. Let me assure you though, with the right approach, this upgrade process is much simpler than you think.

 

I recently worked with a client that actively maintained a large AngularJS application and had made the decision to upgrade to Angular. I was tasked with implementing a solution that allowed us to upgrade in small chunks. The obvious choice was to use the UpgradeModule released by the Angular team. This allows you to run an AngularJS/Angular hybrid, meaning both frameworks are running at the same time, allowing you to upgrade on a piece by piece basis.

 

Implementing UpgradeModule was easy. The problem was getting the front end build system working with this hybrid approach. There seemed to be a lack of documentation online about setting up a hybrid app with WebPack. The official docs use SystemJS, so that was out of the question. Some blogs I found used WebPack, but they assumed your AngularJS app was already completely componentized (using the 1.5 component API) and others assumed you were writing ES2015. Neither of these things were true for the client’s application.

 

This article is intended to help other developers like me, running an AngularJS application, using controllers instead of components, and wanting to implement WebPack to build and serve the hybrid app. Let’s get started.

 

For demonstration purposes, I created a very small AngularJS application following John Papa’s best practices guide that shows the hybrid upgrade process. Check it out here:

 

https://github.com/raymer/angularHybrid

 

First, let’s get the application to run as a hybrid with WebPack.

 

First things first. From inside the angularHybrid directory, run “npm install” and “bower install” to get packages. Take a look at package.json to see what we need, it’s mostly the @angular packages, rxjs, zone.js and then webpack and its dependencies.

 

We need to bootstrap the app manually now.

 

Remove ng-app from index.html

Create a file called “main.ts” in the /app/ directory and bootstrap like so:

main.ts

import { platformBrowserDynamic } from '@angular/platform-browser-dynamic';
import { UpgradeModule } from '@angular/upgrade/static';
import { AppModule } from './app.module';

platformBrowserDynamic().bootstrapModule(AppModule).then(platformRef => {
  const upgrade = platformRef.injector.get(UpgradeModule) as UpgradeModule;
  upgrade.bootstrap(document.documentElement, ['myApp']);
});

 

Next, let’s create the other entry files WebPack needs in the same /app/ directory.

 

vendor.ts

// Angular
import '@angular/platform-browser';
import '@angular/platform-browser-dynamic';
import '@angular/core';
import '@angular/common';
import '@angular/http';
import '@angular/router';

// RxJS
import 'rxjs';

// Other vendors for example jQuery, Lodash or Bootstrap
// You can import js, ts, css, sass, ...

polyfills.ts

import 'core-js/es6';
import 'core-js/es7/reflect';
require('zone.js/dist/zone');

if (process.env.ENV === 'production') {
  // Production
} else {
  // Development and test
  Error['stackTraceLimit'] = Infinity;
  require('zone.js/dist/long-stack-trace-zone');
}

 

Next, ensure you have the following script in your package.json:

 

"scripts": {
  "start": "webpack-dev-server --inline --progress --port 8000"
}

 

Lastly, ensure you have tsconfig.json, webpack.config.js and helpers.js files in the root of your app (same level as package.json).

Snag the ones from the angularHybrid app Github page linked above.

 

You’re now ready to run a hybrid. In the /app/ folder, run “npm start”. You should see WebPack compile and start the server. Navigate to http://localhost:8000 to see the app running. Congrats! The infrastructure part of the journey is over, you’re ready to write some code!

Porting a controller to Angular

We are going to port barController.js to Angular. Since this controller has a dependency on barService, I’m going to show how to keep the existing AngularJS implementation of barService in the Angular version of barController.

 

First, we port the controller over to an Angular component

import { Component, OnInit, Inject } from '@angular/core';

@Component({
  selector: 'bar-component',
  templateUrl: 'bar.component.html'
})

export class BarComponent implements OnInit {
  message: string;
  title = 'bar';

  constructor(@Inject('barService') private barService: any) { }

  ngOnInit(): void {
    this.barService.getData().then(result => {
      this.message = result.data.message;
    });
  }
}

 

Next, we create a new Angular module for our bar functionality. Check out bar.module.ts in the /bar/ directory. The relevant section to allow us to use our existing bar service is:

 

export function barServiceFactory(i: any) {
  return i.get('barService');
}

const barServiceProvider = {
  provide: 'barService',
  useFactory: barServiceFactory,
  deps: ['$injector']
};

 

What this does, is tell Angular that when we inject anything with the name ‘barService’, we want the injector to give us the AngularJS version.

 

We then provide it to the bar module:

  providers: [  
    barServiceProvider    
  ],

 

The last step is to create a hybrid router, that can handle both AngularJS and Angular routes.

 

In app.module.ts we will create a hybrid url handling strategy that we provide to the module.

 

class HybridUrlHandlingStrategy implements UrlHandlingStrategy {
  // use only process the `/bar` url
  shouldProcessUrl(url: UrlTree) {
    return url.toString().startsWith('/bar');
  }
  extract(url: UrlTree) { return url; }
  merge(url: UrlTree, whole: UrlTree) { return url; }
}

providers: [
  { provide: UrlHandlingStrategy, useClass: HybridUrlHandlingStrategy }       
],

 

Then we change our app.component.ts to inject both routing components.

import { Component } from '@angular/core';

@Component({
  selector: 'my-app',
  template: `
    <router-outlet></router-outlet>
    <div ng-view></div>
  `,
})

export class AppComponent { }

 

That’s it. Now when we navigate to /foo, we will hit the AngularJS controller for foo, and when we navigate to /bar, we will hit the Angular bar component.

 

In this post, I have shown the process of upgrading an AngularJS controller to Angular, utilizing the existing AngularJS dependencies. With this strategy of upgrading an application, we can port one module at a time, making the upgrade process much simpler.

 

I welcome your comments on this process and any questions you may have.
