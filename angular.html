Angular Route Resolvers and RxJS

Jordan Koidahl Development

With the introduction of Reactive Extensions in RxJS, Angular has provided developers with a number new powerful tools for working with all things asynchronous. While RxJS allows for developers to effectively manage asynchronous processes, using reactive extensions in angular is not without its hazards. Below is the description and resolution of an issue I recently encountered. To help illustrate, I will borrow the superhero domain from the Angular Tutorial.

Use Case

This scenario involves a web application which manages comic book superheroes. The web-app and is laid out such that users create/edit Heroes in the context of a universe (e.g. DC, Marvel, etc). The particular user story we are trying to implement is:

From the universe detail view, a user can create/edit a superhero that will be associated with the current universe. Users will be able to assign one to many authors to a given superhero. This particular problem occurred during the initialization of the HeroDetail component.

Technical details

Our web application makes use of the Angular route resolver to pre-load the “Universe” object when accessing the hero detail route. In the event of a create, we will instantiate a new Hero and assign the current universe context and load a master author collection for users to assign to this new hero. In the event of an edit, retrieve the Hero, retrieve the master author collection and in order for the author list form input to load selected authors, update the hero’s author collection by assign matching on author’s id.

First attempt:

ngOnInit() {
  this.http.get('authorListEndpoint').map(res => res.json())
    .subscribe( list => this.masterAuthorList = list);
  const heroId: string = this.route.snapshot.paramMap.get('heroId');
  this.route.data.switchMap( (data) => {
    // Create hero
    if ( ! heroId) {
      let universe = data.universe;
      this.hero = new Hero(universe);
    // Edit hero
    } else {
      this.http.get('heroEndpoint')
      .map( res => res.json())
      .subscribe(hero => {
        this.hero = hero;
        const herosAuthorIds = this.hero.authorList.map( author => author.id);
        this.hero.authorList = this.masterAuthorList.filter( author => {
          herosAuthorIds.contains(author.id) });
      });
    }
  });
}

 

This works fine for the initialization of a new Hero Detail form but not for editing an existing hero. Specifically, the Hero Detail form would load without any of the authors selected. To fix this I attempted to use the forkJoin RxJS operator which would run, the loading of the hero and loading of the author list in parallel before combining the results.

 

Second Pass:


ngOnInit() {
  let authorList$ = this.http.get('authorListEndpoint').map(res => res.json());
  const heroId: string = this.route.snapshot.paramMap.get('heroId');
  let heroLoad$ = this.route.data.switchMap( (data) => {
    if ( ! heroId) {
      let universe = data.universe;
      return Observable.from( new Hero(universe) );
    }
    return this.http.get('heroEndpoint').map( res => res.json());
  });
  Observable.forkJoin([authorList$, heroLoad$]).subscribe( results => {
    this.masterAuthorList = results[0];
    this.hero = results[1];
    const herosAuthorIds = this.hero.authorList.map( author => author.id);
    this.hero.authorList = this.masterAuthorList.filter( author => {
              herosAuthorIds.contains(author.id) });
  });
}

 

Unfortunately, the forkJoin’s subscribe block would not execute. This is because ForkJoin expects inputs to complete before the subscribe logic is run.  In this case, the observable returned by http.get is completes and the observable returned by ActivatedRoute.data does not. To fix this issue I replaced forkJoin with combineLatest (see fixed implementation below) and the form loads correctly with the hero’s authors selected.

 

Lessons Learned

  1. Unless you have only Angular http observables (which complete after a single event), don’t use ForkJoin, ForkJoin only works with completed observables
  2. ActivatedRoute observables do not complete and therefore will not trigger a forkJoin subscription event.
  3. CombineLatest > ForkJoin

Here is the fixed Implementation:
hero-detail.component.ts


import { Component } from '@angular/core';
import { Http } from '@angular/http';
import { Observable } from 'rxjs/Obervable';
import { Hero } from '../model/hero';
@Component
export class HeroDetailComponent implements OnInit {
  hero: {};
  masterAuthorsList: Array = [];
  constructor ( private route: ActivatedRoute ) { }
  ngOnInit() {
    const heroId: string = this.route.snapshot.paramMap.get('heroId');
    let authorList$ = this.http.get('authorListEndpoint').map( res => res.json() );
    let loadHero$ = this.route.data.switchMap( (data) => {
      //create hero
      if ( ! heroId) {
        let newHero = new Hero(data.universe);
        return Observable.from( [ newHero ] );
      }
      return this.http.get( 'heroListEndpoint' + heroId).map(res => res.json());
    });
    Observable.combineLatest( [ authorList$, loadHero$ ] ).subscribe( results => {
      this.masterAuthorList = results[0];
      this.hero = results[1];
      const herosAuthorIds = this.hero.authorList.map( author => author.id);
      this.hero.authorList = this.masterAuthorList.filter( author => {
        herosAuthorIds.contains(author.id) });
      });
    }
}
